# Contributing

Thanks for helping improve Awesome ServiceNow. The goal is to maintain a useful, high-quality Awesome List for the ServiceNow ecosystem.

## Quality Bar

- Prefer authoritative, maintained, and practically useful resources.
- Avoid marketing-only pages, gated lead-generation pages, thin content, and abandoned projects.
- Do not add duplicate links.
- Keep descriptions concise and factual.
- Use Markdown only.
- Keep entries alphabetized within sections where practical.

## Pull Requests

- One resource per pull request is preferred.
- Include the resource name, direct URL, and a short description.
- Verify the link is active before submitting.
- Explain why the resource belongs in the list if it is not obviously authoritative.
- Add the entry to the most relevant existing section instead of creating a new section for one item.

## Entry Format

Use this format:

```markdown
- [Resource Name](https://example.com/) - One concise sentence describing the resource.
```

## What To Include

Good candidates include:

- Official ServiceNow documentation, product pages, learning resources, and community resources.
- Active open-source ServiceNow tools, integrations, examples, and developer utilities.
- High-quality blogs, videos, podcasts, newsletters, and community creator resources.
- Practical architecture, DevOps, observability, AI, automation, and platform engineering resources.
- Community events, conferences, user groups, and learning programs.

## What Not To Include

Do not submit:

- Duplicate links already present in the list.
- Dead links or resources that require unclear access to evaluate.
- Pure sales pages with no educational or practical value.
- Low-effort AI-generated content, scraped content, or copied documentation.
- Personal affiliate links, tracking links, or shortened URLs.
- Resources unrelated to ServiceNow or the Now Platform.

## Link Checking

Before opening a pull request:

1. Open the link in a browser or run a link checker.
2. Confirm the page is still active and relevant.
3. Confirm the URL is direct and not a redirect, short link, or tracking URL when a cleaner URL is available.
4. Search the README for the URL to avoid duplicates.

## Style

- Use sentence case for descriptions.
- Do not end descriptions with marketing slogans.
- Keep each description to one sentence.
- Prefer official names for products, projects, and creators.
- Use HTTPS URLs where available.

## Code of Conduct

By participating, you agree to follow the [Code of Conduct](CODE_OF_CONDUCT.md).
